# Setup

One-time DSM setup, then wire credentials into your shell. All of it is clickthrough in DSM except the last step.

## 1. Dedicated DSM user

DSM → Control Panel → User & Group → Create.

| Setting | Value |
|---|---|
| Username | `claude-mcp` |
| Password | strong random, generated by DSM, captured straight into your password manager |
| Disallow password change | yes |
| Password never expires | yes |
| User group | `users` **and** `administrators` (see below) |
| 2-Factor Authentication | enable, and capture the TOTP **secret** (the base32 string, not the 6-digit code) |

The TOTP secret is the part people miss. DSM shows it once when you enable 2FA. The CLI generates codes from it, so without it there is no unattended login.

### Why `administrators`

DSM 7 gates its admin APIs (`SYNO.Core.Package.*`, `SYNO.SecurityAdvisor.*`, `SYNO.Core.User.*`, `SYNO.Core.Share`) on `administrators` membership. There is no selective grant. Control Panel → Application Privileges covers only end-user services like File Station and SMB, not the admin apps, so restricting those there does not reduce what this account can reach. An admin has share access regardless.

So the account is an admin, and the controls that actually bound it are:

1. **The password is never typed.** Generate it in DSM, store it in your password manager, feed it to the CLI as an env var or a file.
2. **2FA enforced.** Password alone gets no login.
3. **SSH off** unless you need it (Control Panel → Terminal & SNMP). Admin group implies SSH eligibility, so the service being off is what stops it.
4. **Network reachability.** DSM's own port (5001) should be restricted to devices you control, by tailnet ACL or firewall.

Residual risk is full DSM compromise if your secret store leaks and the NAS is reachable. Fine for personal use, but know which controls are load-bearing.

## 2. Credentials

The CLI reads credentials from the environment, or from files if you'd rather not put secrets in env vars. Nothing is built in to any particular secret manager. See `src/auth.ts` for the exact resolution order.

| Variable | File form | What |
|---|---|---|
| `DSM_PASSWORD` | `DSM_PASSWORD_FILE` | the account password |
| `DSM_TOTP_SECRET` | `DSM_TOTP_SECRET_FILE` | the base32 TOTP secret |

Plus `DSM_BASE_URL` (e.g. `https://nas.local:5001`) and, optionally, `DSM_USER` if you didn't name the account `claude-mcp`.

Populate them however you like. A launcher that fetches from your password manager works (`op run`, `sops exec-env`), as does a `chmod 600` file, as does plain env for a throwaway shell.

For a router target, the same pattern with the `SRM_` prefix, plus `SRM_BASE_URL`. The router client is read-only and needs a dedicated SRM admin (Control Panel → User → Grant administrator privilege); a normal user gets code 402 at login.

## 3. Install

```sh
npm install -g .     # puts `syno` on your PATH
syno --help
```

## 4. Verify

```sh
syno status
```

Expect JSON with model, DSM version, uptime and load. If login fails, DSM answers with a code rather than a message: 400 is a bad password, 404 is a rejected 2FA code (usually a wrong or stale TOTP secret), 402 means the account isn't an admin.

The first run writes a session file to `~/.local/state/syno/`, so subsequent commands reuse the login instead of burning a TOTP code each time.

## 5. Optional: DSM notification email

DSM → Control Panel → Notification → Email. DSM then emails you when packages have updates, which is a useful backstop to `syno updates`.

## Uninstall

```sh
npm uninstall -g synology-cli
rm -rf ~/.local/state/syno
```

Then delete the DSM user in Control Panel and remove the credentials from wherever you stored them.
